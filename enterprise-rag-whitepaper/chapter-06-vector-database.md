## 1. 为什么需要向量数据库

### 1.1 概念定义

向量数据库（Vector Database）是一种专门设计用于存储、索引和检索高维向量嵌入（Embedding）的数据库系统。它通过近似最近邻（ANN, Approximate Nearest Neighbor）搜索算法，实现大规模高维向量的高效相似度检索。

**核心数据结构 -- 向量嵌入**：

```
文本: "今天天气真好，适合出去散步"
  |
  v  Embedding Model (text-embedding-3-large)
  |
  v
向量: [0.023, -0.145, 0.891, ..., 0.032]   # 3072维 float32
```

### 1.2 向量数据库 vs 传统数据库 vs 内存计算

#### 1.2.1 与传统关系型数据库的对比

| 维度 | 传统关系型数据库 (MySQL/PostgreSQL) | 向量数据库 (Milvus/Qdrant) |
|------|-------------------------------------|-----------------------------|
| **数据模型** | 结构化表格，行与列 | 非结构化嵌入向量 + 元数据 |
| **查询方式** | 精确匹配 (WHERE id = 5) | 相似度搜索 (向量距离最近) |
| **索引结构** | B-Tree / Hash / GIN | HNSW / IVF / PQ |
| **查询语义** | 确定性的，布尔逻辑 | 非确定性的，模糊匹配 |
| **典型QPS** | 100K+ (点查) | 1K-50K (向量搜索) |
| **精度** | 100% 精确 | 95%-99.9% 近似召回 |

**为什么传统数据库不能胜任向量搜索？**

```
场景：在 1000 万条文档中查找与用户问题最相似的 10 个段落

方案A：MySQL (LIKE / FULLTEXT)
  SELECT * FROM documents WHERE content LIKE '%今天天气%';
  问题：仅能做关键词匹配，无法理解语义。语义相近但词汇不同时完全失效。

方案B：PostgreSQL + pgvector (暴力搜索)
  SELECT * FROM documents ORDER BY embedding <=> query_vector LIMIT 10;
  问题：无索引情况下，需要遍历全部 1000 万条向量，计算 1000 万次余弦相似度。
  单条查询耗时：1000万 * 3072维 * 2 FLOPs = ~61.4 GFLOPS → 约 5-30 秒

方案C：专用向量数据库 (HNSW索引)
  构建 HNSW 图索引（离线），查询时沿图搜索，只需访问 ~log(N) 个节点。
  单条查询耗时：~100 * 3072维 * 2 FLOPs = ~0.6 MFLOPS → 约 0.5-5 毫秒
```

#### 1.2.2 与内存 NumPy/FAISS 的对比

| 维度 | 内存方案 (NumPy/FAISS in-memory) | 向量数据库 (Milvus/Qdrant) |
|------|----------------------------------|-----------------------------|
| **持久化** | 无，进程退出数据丢失 | 原生支持，WAL + 快照 |
| **并发能力** | 单机单进程 | 多副本读写分离，支持高并发 |
| **增量更新** | 需全量重建索引 | 支持实时插入/删除/更新 |
| **横向扩展** | 不支持 | 分片 + 副本，支持水平扩展 |
| **元数据过滤** | 需自行实现 | 内置标量过滤 + 向量搜索 |
| **监控运维** | 无 | Prometheus/Grafana 监控 |
| **数据一致性** | 无担保 | ACID / 最终一致性可选 |
| **权限管控** | 无 | RBAC、多租户隔离 |
| **数据规模** | 受限于单机内存 | PB级，分布式存储 |
| **故障恢复** | 需手动重建 | 自动故障转移、数据恢复 |

**代码对比 -- 内存方案 vs 向量数据库**：

```python
# ============ NumPy 内存方案 ============
import numpy as np

# 数据全部加载到内存
vectors = np.load("embeddings.npy")  # 1000万 x 768维, ~30GB
query = get_embedding("用户问题")

# 暴力搜索 -- O(N*d)
distances = np.linalg.norm(vectors - query, axis=1)
top_k = np.argsort(distances)[:10]

# 问题：
# 1. 30GB 内存常驻，无法释放
# 2. 进程重启后需要重新加载（冷启动 2-5 分钟）
# 3. 无法增量更新 -- 新增 1 条向量需要重新加载全量
# 4. 无法多进程共享 -- 每个进程 30GB 内存开销
# 5. 无并发控制 -- 多线程读写存在竞态
# 6. 无可用性保障 -- 机器宕机数据全丢


# ============ 向量数据库方案 ============
from pymilvus import Collection

collection = Collection("documents")
collection.load()  # 数据按需从磁盘/内存映射

# ANN 搜索 -- O(log N) 近似
results = collection.search(
    data=[query_vector],
    anns_field="embedding",
    param={"metric_type": "COSINE", "params": {"ef": 128}},
    limit=10,
    expr="category == '技术文档'",  # 标量过滤 + 向量搜索
    output_fields=["title", "author", "created_at"]
)

# 优势：
# 1. 内存用量：索引结构而非全量数据常驻（HNSW 约 1.5x 原始数据大小）
# 2. 持久化：数据写入即持久化，进程重启毫秒级恢复
# 3. 增量：插入新向量后索引自动更新，毫秒级可见
# 4. 并发：多副本读写分离
# 5. 扩展：增加节点即可水平扩展至百亿级别
```

### 2.2 FAISS (Facebook AI Similarity Search)

#### 2.2.1 概述与架构

FAISS 是 Meta 开源的向量相似度搜索库，严格来说是一个**向量索引引擎**而非完整数据库。它提供了丰富的索引算法，支持 GPU 加速，是许多向量数据库的底层索引引擎。

```
FAISS 架构示意

┌────────────────────────────────────────────────────┐
│                    FAISS Library                    │
│                                                    │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────┐ │
│  │  IndexFlat   │  │   IndexIVF   │  │IndexHNSW │ │
│  │  (暴力搜索)   │  │ (倒排索引)   │  │ (图索引) │ │
│  └──────────────┘  └──────────────┘  └──────────┘ │
│  ┌──────────────┐  ┌──────────────┐               │
│  │   IndexPQ    │  │  IndexIVFPQ  │               │
│  │ (乘积量化)   │  │ (IVF+PQ组合) │               │
│  └──────────────┘  └──────────────┘               │
│                                                    │
│  ┌──────────────────────────────────────────────┐  │
│  │              GPU 加速层                        │  │
│  │  GpuIndexFlat  |  GpuIndexIVF  |  GpuCloner  │  │
│  └──────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────┘
```

#### 2.2.2 核心特性

| 特性 | 说明 |
|------|------|
| **类型** | C++ 库，非完整数据库 |
| **索引类型** | Flat, IVF, HNSW, PQ, IVFPQ, LSH, NSG 等 20+ 种 |
| **GPU 支持** | 原生支持，多 GPU 并行，O(100x) 加速 |
| **内存使用** | 依索引类型，HNSW 约 1.5x 原始向量大小 |
| **查询速度** | 单机可达 10K-100K QPS (GPU) |
| **分布式** | 无内建支持，需自行实现分片 |
| **持久化** | 支持磁盘存储与内存映射 |
| **增量更新** | 部分索引支持 (HNSW 支持，IVF 需重建) |
| **过滤能力** | 仅 IDFilter，需自行实现标量过滤 |
| **企业成熟度** | 中等 -- 非数据库，需要封装层 |

#### 2.2.3 索引类型深度对比

```
FAISS 索引选择决策树

需要精确搜索？
  ├── 是 → IndexFlatL2 / IndexFlatIP (数据量 < 1M)
  └── 否 → 内存是否受限？
            ├── 否 → 对召回率要求高？
            │         ├── 是 → IndexHNSWFlat (M=16, efConstruction=200)
            │         └── 否 → IndexIVFFlat (nlist=sqrt(N))
            └── 是 → IndexIVFPQ (压缩比可达 8x-16x)
```

#### 2.2.4 优缺点总结

| 优势 | 劣势 |
|------|------|
| 算法最全面，20+ 种索引 | 非完整数据库，无 CRUD API |
| GPU 加速成熟度高 | 无内置分布式 |
| 社区活跃，文档丰富 | 需要自行封装服务层 |
| 轻量级，无外部依赖 | 无元数据管理能力 |
| 索引构建速度领先 | 增量更新不便 |

### 2.4 Qdrant

#### 2.4.1 概述与架构

Qdrant 是使用 Rust 语言编写的高性能向量数据库，以单文件二进制分发，注重极致性能和开发体验。

```
│  │   预过滤 (Pre-filter) → 向量搜索 → 后过滤  │  │
│  │   标量索引 + 向量索引 混合查询             │  │
│  └───────────────────────────────────────────┘  │
├─────────────────────────────────────────────────┤
│              向量存储引擎 (Segment)              │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐         │
│  │ Segment │  │ Segment │  │ Segment │  ...    │
│  │  (R/O)  │  │  (R/O)  │  │  (R/W)  │         │
│  └─────────┘  └─────────┘  └─────────┘         │
│       │            │            │               │
│  ┌────▼────────────▼────────────▼──────────┐   │

├─────────────────────────────────────────────────┤

└─────────────────────────────────────────────────┘
```

#### 2.4.2 核心特性

| 特性 | 说明 |
|------|------|
| **语言** | Rust，单二进制部署 |
| **索引类型** | HNSW (主要) |
| **量化** | Scalar Quantization, Product Quantization, Binary Quantization |
| **过滤能力** | 强大的 Payload Filtering (支持嵌套、全文) |
| **分布式** | 原生支持集群模式 (Raft 共识) |
| **高可用** | 多副本 + 自动 Leader 选举 |
| **内存使用** | 量化模式下内存占用极低 (1/4 ~ 1/16) |
| **查询速度** | 单节点 5K-40K QPS (取决于量化方式) |
| **数据一致性** | Raft 强一致 |
| **客户端** | Python, Rust, Go, Java, .NET, TypeScript |
| **企业成熟度** | 中高 -- Qdrant Cloud 商业支持 |

#### 2.4.4 优缺点总结

| 优势 | 劣势 |

## 3. 索引算法深度剖析

### 3.1 索引算法全景

```
索引算法演化树

                        Flat (暴力搜索)
                        O(N*d)
                          │
          ┌───────────────┼───────────────┐
          ▼               ▼               ▼
     空间划分          量化压缩          图索引
          │               │               │
    ┌─────┴─────┐   ┌─────┴─────┐   ┌─────┴─────┐
    │    IVF    │   │    PQ     │   │   HNSW    │
    │ O(N/nlist)│   │  d→d/m    │   │ O(logN)   │
    └─────┬─────┘   └─────┬─────┘   └─────┬─────┘
          │               │               │
          └───────┬───────┘               │
                  ▼                       ▼
              IVFPQ                  DiskANN
           (IVF+PQ组合)           (Vamana图+磁盘)
```

### 3.2 Flat -- 暴力搜索

#### 3.2.1 原理

Flat 索引不做任何预处理，查询时直接计算查询向量与所有数据向量的距离，返回 Top-K 最近邻。

```
Flat 搜索流程

    查询向量 q [d维]
        │
        ▼
  ┌─────────────────────────────────────┐
  │  for i in 1..N:                     │
  │    distance[i] = dist(q, vectors[i])│
  │                                     │
  │  top_k = argsort(distances)[:K]     │
  └─────────────────────────────────────┘
        │
        ▼
   返回 K 个最近邻

时间复杂度：O(N*d) -- N 为向量数量，d 为维度
空间复杂度：O(N*d) -- 存储原始向量
```

#### 3.2.2 性能特征

| 维度 | 值 |
|------|-----|
| **构建时间** | 0 (无需构建) |
| **搜索复杂度** | O(N*d) |
| **搜索100万向量** | ~50-200ms (CPU), ~1-5ms (GPU) |
| **搜索1000万向量** | ~500ms-2s (CPU), ~10-50ms (GPU) |
| **搜索1亿向量** | ~5-20s (CPU), ~100-500ms (GPU) |
| **内存占用** | N*d*4 字节 |
| **召回率** | 100% (精确结果) |
| **适用规模** | < 10万向量 (CPU), < 1000万 (GPU) |

**使用场景**: 原型验证、基准测试、小规模数据集 (<100K)

---

### 3.3 IVF (Inverted File Index) -- 倒排索引

#### 3.3.1 核心原理

IVF 通过聚类将向量空间划分为多个区域，查询时仅搜索最近的几个区域。

```
IVF 索引构建流程

Step 1: K-Means 聚类中心学习

  原始向量空间 (N 个向量)
  ┌─────────────────────────────┐
  │  ·  ··   ··    ·  ··  ···  │
  │    ···  ·  ·    ·  ····    │     K-Means (nlist 个聚类中心)
  │  ·  ·· ·  ·  ··   ·  ···  │  ────────────────────────►
  │     ··  ·· ·  ··  ·  ··   │
  │  ··  ··   ·  ·· ···  · ·  │
  └─────────────────────────────┘

  ┌─────────────────────────────┐
  │  ┌──┐      ┌──┐            │
  │  │C1│ ...  │C4│  nlist=4   │
  │  └──┘      └──┘            │
  │  ┌──┐      ┌──┐            │
  │  │C2│ ...  │Cn│            │
  │  └──┘      └──┘            │
  └─────────────────────────────┘


Step 2: 向量分配到最近的聚类中心 (倒排列表构建)

  聚类中心 C1      聚类中心 C2      聚类中心 C3      聚类中心 C4
  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
  │ vector_1 │   │ vector_5 │   │ vector_3 │   │ vector_7 │
  │ vector_2 │   │ vector_8 │   │ vector_4 │   │ vector_9 │
  │ vector_6 │   │          │   │          │   │ vector_10│
  └──────────┘   └──────────┘   └──────────┘   └──────────┘
    倒排列表1       倒排列表2       倒排列表3       倒排列表4


Step 3: 查询时搜索流程

  查询向量 q
      │
      ▼
  计算 q 与 nlist 个聚类中心的距离
      │
      ▼
  选择最近的 nprobe 个聚类中心 (nprobe << nlist)
      │
      ▼
  ┌─────────────────────────┐
  │ 仅搜索选中的倒排列表    │
  │ 候选集大小 ≈ N * nprobe/nlist  │
  └─────────────────────────┘
      │
      ▼
  对候选集进行精确距离计算，返回 Top-K
```

#### 3.3.2 关键参数

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| **nlist** | 聚类中心数量 | sqrt(N) ~ 4*sqrt(N) |
| **nprobe** | 查询时搜索的聚类数 | 1 ~ nlist/10 |

**nprobe 对召回率的影响 (经验数据，nlist=1024, N=100万)**:

| nprobe | 搜索范围 | 召回率@10 | 延迟 | 
|--------|---------|-----------|------|
| 1 | 0.1% | 60-70% | 0.5ms |
| 4 | 0.4% | 85-90% | 1ms |
| 16 | 1.6% | 95-98% | 3ms |
| 64 | 6.25% | 99-99.5% | 10ms |
| 256 | 25% | 99.9%+ | 35ms |

#### 3.3.3 性能特征

| 维度 | 值 |
|------|-----|
| **构建时间** | O(N * nlist * d * iter) -- K-Means 迭代 |
| **搜索复杂度** | O(nprobe * N/nlist * d) -- 仅搜索部分区域 |
| **内存占用** | N*d*4 + nlist*d*4 (原始向量 + 聚类中心) |
| **召回率** | 90%-99.9% (取决于 nprobe) |
| **适用规模** | 100万 ~ 1亿 |

### 3.4 HNSW (Hierarchical Navigable Small World)

#### 3.4.1 核心原理

HNSW 是目前实践中性能最优的 ANN 算法之一，通过构建多层图结构实现 O(logN) 的近似搜索。

```
HNSW 多层图结构

          Layer 2 (顶层，最稀疏)
          ┌───────────────────────────────────┐
          │          ● ───────── ●            │
          │         /              \          │
          │        ●                ●         │   进入点 (Entry Point)
          │                                 │   ↓ ep
          └───────────────────────────────────┘
                        │
                        ▼
          Layer 1 (中间层)
          ┌───────────────────────────────────┐
          │      ● ─── ● ─── ● ─── ●         │
          │     / \    / \    / \    / \      │
          │    ●   ●──●   ●──●   ●──●   ●    │
          │     \ /    \ /    \ /    \ /      │
          │      ● ─── ● ─── ● ─── ●         │
          └───────────────────────────────────┘
                        │
                        ▼
          Layer 0 (底层，最稠密)
          ┌───────────────────────────────────┐
          │   ●──●──●──●──●──●──●──●──●──●  │
          │   |\/|\/|\/|\/|\/|\/|\/|\/|\/|  │
          │   ●──●──●──●──●──●──●──●──●──●  │
          │   |\/|\/|\/|\/|\/|\/|\/|\/|\/|  │
          │   ●──●──●──●──●──●──●──●──●──●  │
          │   包含所有 N 个向量节点            │
          └───────────────────────────────────┘

图结构特性：
- 层数 L = floor(-ln(uniform(0,1)) * m_L)，其中 m_L = 1/ln(M)
- 每层节点期望数量：N / M^layer (M 默认16)
- Layer 0 包含所有节点
- 每增加一层，节点数减少约 1/M
```

#### 3.4.2 HNSW 图构建算法

```
HNSW 构建算法伪代码

INSERT(hnsw, q, M, Mmax, efConstruction, m_L)
  Input: 多层图 hnsw, 新元素 q, 连接数 M, 最大连接数 Mmax,
         构建时搜索宽度 efConstruction, 层标准化因子 m_L
  
  Step 1: 确定 q 所在的最高层 l
    l = floor(-ln(random(0,1)) * m_L)
  
  Step 2: 获取入口点 ep (顶层任意点)，获取 ep 的最高层 L
    if hnsw 为空:
      将 q 插入所有层，q 成为入口点，返回
  
  Step 3: 从顶层 L 向下贪心搜索到第 l+1 层
    for lc = L downto l+1:
      W = SEARCH-LAYER(q, ep, ef=1, lc)  # 仅找最近邻
      ep = W 中最近的节点
  
  Step 4: 从第 l 层向下到第 0 层，构建连接
    for lc = min(L, l) downto 0:
      W = SEARCH-LAYER(q, ep, efConstruction, lc)
      neighbors = SELECT-NEIGHBORS(q, W, M, lc)
      在第 lc 层添加 q 与 neighbors 的双向边
      for each n in neighbors:
        如果 n 的边数 > Mmax:
          收缩 n 的邻域 (SELECT-NEIGHBORS 启发式)


SELECT-NEIGHBORS(q, candidates, M, lc)
  Input: 新节点 q, 候选邻居集 candidates, 目标连接数 M
  Output: M 个经过启发式选择的邻居
  
  简单模式 (FAISS 实现):
    return candidates 中距离 q 最近的 M 个
  
  启发式模式 (HNSW 论文原版):
    R = []           # 结果集
    W = candidates   # 候选工作集
    while W 非空 and |R| < M:
      e = W 中距离 q 最近的元素
      R.append(e)
      如果 e 比 q 到 R 中已有元素的任一距离更近:
        从 W 中移除 e
      # 否则 e 被保留为有效连接
    return R
```

#### 3.4.3 HNSW 搜索算法

```
HNSW 搜索算法

SEARCH-LAYER(q, ep, ef, lc)
  Input: 查询向量 q, 入口点 ep, 搜索宽度 ef, 当前层 lc
  Output: ef 个最近邻
  
  Step 1: 初始化
    v = ep          # 当前访问节点
    C = {ep}        # 候选集 (最小堆，按距离排序)
    W = {ep}        # 最近邻结果集 (最大堆，按距离排序)
  
  Step 2: 贪心遍历 (在同层内)
    while C 非空:
      c = C 中距离 q 最近的元素 (弹出)
      f = W 中距离 q 最远的元素
      
      if dist(c, q) > dist(f, q):
        break       # 所有候选元素都比当前结果远，终止
      
      for each neighbor e of c:
        if e 未被访问过 且 dist(e, q) < dist(f, q):
          将 e 加入 C
          将 e 加入 W
          如果 |W| > ef:
            从 W 中移除最远元素 (保持 W 大小为 ef)
  
  Step 3: 返回 W

查找流程示意 (ef=3):

  起始点 ep ●
           / \
          ●───●  <-- 搜索沿边展开
         /     \
        ●───────●  <-- 候选集动态更新
       /         \
      ●─────●─────●  <-- 最终 Top-3 结果
```

### 3.5 PQ (Product Quantization) -- 乘积量化

#### 3.5.1 核心原理

PQ 通过子空间聚类将高维向量压缩为短编码，大幅减少存储和计算开销。

```
PQ 编码过程

Step 1: 向量切分

  原始向量: [v1 v2 v3 v4 v5 v6 v7 v8 | v9 v10 v11 v12 v13 v14 v15 v16]
             \_________子向量1________/ \___________子向量2___________/
  
  对于一个 d 维向量，将其切分为 m 个子向量，每个子向量维度为 d* = d/m


Step 2: 子空间聚类与码书生成

  子空间1 (维度 1-8)                   子空间2 (维度 9-16)
  ┌─────────────────┐                 ┌─────────────────┐
  │ 聚类中心 c1_1   │                 │ 聚类中心 c2_1   │
  │ 聚类中心 c1_2   │                 │ 聚类中心 c2_2   │
  │ 聚类中心 c1_3   │                 │ 聚类中心 c2_3   │
  │     ...         │                 │     ...         │
  │ 聚类中心 c1_256 │                 │ 聚类中心 c2_256 │
  └─────────────────┘                 └─────────────────┘
       码书1                                码书2

  每个子空间有 k = 256 个聚类中心 (8 bits 表示)


Step 3: 向量编码

  原始向量: [0.23, -0.45, 0.67, ..., -0.12]  (d * 4 字节)
      │
      ▼ 子向量划分 + 最近邻聚类中心查找
      │
  PQ 编码: [idx1 | idx2 | ... | idx_m]
           每个 idx 为 0-255，用 8 bits 表示
  
  压缩比: (d * 4) / (m * 1) = 4d / m bytes
         如果 d=768, m=96, 则 3072字节 → 96字节, 压缩比 32x


Step 4: 非对称距离计算 (ADC -- Asymmetric Distance Computation)

  查询向量 q 不编码，数据向量用 PQ 编码
  距离 dist(q, x) ≈ sqrt(sum_{j=1..m} dist(q_j, c_j[idx_j])^2)
  
  预计算查询向量到所有码书中心的距离表:
  lookup_table[j][k] = dist(q_j, c_j_k)   # 大小: m * 256
                                         # 例如: 96 * 256 * 4 = 96KB
  
  对任意编码后的向量，距离计算变为:
  dist_approx = 0
  for j in 1..m:
    dist_approx += lookup_table[j][pq_code[j]]
  
  从 O(d) 降到 O(m) 次查表操作!
```

#### 3.5.2 性能特征

| 维度 | 值 |
|------|-----|
| **构建时间** | 需 K-Means 训练码书 (一次性) |
| **内存占用** | N * m * 1 字节 + m * k * d/m * 4 (编码 + 码书) |
| **压缩比** | 典型 8x-32x (取决于 m 和 k) |
| **搜索速度** | 查表 + 累加，极快 |
| **召回率** | 90%-98% (单独使用, 取决于压缩比) |
| **适用规模** | 内存受限，TB 级数据 |

### 3.7 DiskANN (Disk-based ANN)

#### 3.7.1 核心原理

DiskANN 基于 Vamana 图，将图结构和向量数据存储在 SSD 上，通过最小化磁盘 I/O 实现大规模 ANN 搜索。

```
DiskANN 架构

┌─────────────────────────────────────────────────────┐
│                   内存 (DRAM)                        │
│  ┌───────────────────────────────────────────────┐  │
│  │           PQ 压缩后的码书 + 缓存               │  │
│  │           beam search 的候选队列               │  │
│  │           热点图节点缓存                       │  │
│  └───────────────────────────────────────────────┘  │
│                         ↕ I/O                        │
├─────────────────────────────────────────────────────┤
│                  SSD (NVMe)                         │
│  ┌───────────────────────────────────────────────┐  │
│  │          Vamana 图结构 (邻接表)               │  │
│  │          原始向量数据 (压缩存储)               │  │
│  └───────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘

Vamana 图构建：
  1. 随机初始化图
  2. 对每个节点执行 beam search 找到候选邻居
  3. 使用 Robust Pruning: 
     保留到候选距离最近的 α*R 个邻居中的 R 个
     α >= 1 控制图的稠密度
  4. 确保图直径在 log(N) 量级

搜索：
  1. 从入口点开始 beam search
  2. 每次访问节点时从 SSD 读取其邻接表
  3. 使用 PQ 编码近似距离作为预筛选
  4. 对候选集进行精确重排
```

**性能特征**: 百亿级向量，单机 SSD，约 5-50ms 延迟，90%+ 召回率。

## 4. 近似最近邻搜索数学推导

### 4.1 问题形式化

**精确最近邻 (Exact Nearest Neighbor)**:

给定数据集 D = {x_1, x_2, ..., x_N} 包含于 R^d，查询向量 q 包含于 R^d，距离度量 dist: R^d × R^d → R^+。

精确最近邻搜索结果:
```
NN(q, D) = argmin_{x ∈ D} dist(q, x)
```

Top-K 精确最近邻:
```
KNN(q, D) = {x_{(1)}, x_{(2)}, ..., x_{(K)}} 
其中 x_{(i)} 是第 i 近的向量
```

**近似最近邻 (Approximate Nearest Neighbor)**:

对于 ε > 0，向量 x' 是 q 的 (1+ε)-近似最近邻，当且仅当:
```
dist(q, x') ≤ (1+ε) * dist(q, x*)  # x* 是精确最近邻
```

在实际系统中，我们更关心 **召回率 (Recall)**:
```
Recall@K = |ANN_K(q) ∩ NN_K(q)| / K
即近似Top-K结果与精确Top-K结果的重合比例
```

### 4.2 HNSW 图结构的数学性质

#### 4.2.1 层级分布

HNSW 中每个节点所在的最高层级服从几何分布:
```
P(max_level = l) = (1 - 1/M) * (1/M)^l * N

期望每层节点数:
E[|Layer l|] = N * (1/M)^l

Layer 0:  N 个节点
Layer 1:  ~N/M 个节点
Layer 2:  ~N/M^2 个节点
...
```

`m_L = 1/ln(M)` 确保图直径在 O(log N) 量级。

#### 4.2.2 搜索复杂度分析

```
HNSW 搜索复杂度 (概率分析):

设 M 为每节点连接数，层数 L ≈ log_M(N)

在 Layer lc 的搜索复杂度:
  - 从入口点开始遍历
  - 每步向更近的节点移动
  - 期望访问节点数: O(ef) = O(efSearch)
  - 期望距离计算数: O(ef * M)

总搜索复杂度:
  T_search = sum_{lc=1..L} O(ef * M) = O(log_M(N) * efSearch * M)

由于 M 和 efSearch 是常数:
  T_search = O(log N)  ← 这就是 ANN 能够实现对数时间的数学保证
```

#### 4.2.3 小世界特性

NSW (Navigable Small World) 图具有两个关键属性:

1. **长程连接 (Long-range edges)**: 高层节点稀疏但跨度大，实现快速区域定位
2. **短程连接 (Short-range edges)**: 底层节点密集，保证局部精确搜索

```
搜索过程的数学描述 -- 逐步收敛:

初始距离:        r_0 = dist(q, ep)
第1步后距离:     r_1 ≤ γ * r_0      (γ < 1, 每步指数收敛)
第2步后距离:     r_2 ≤ γ^2 * r_0
...
第t步后距离:     r_t ≤ γ^t * r_0

当 r_t ≤ 目标区域半径时搜索终止
期望步数 t ≈ log_{1/γ}(r_0 / r_target) = O(log N)
```

### 4.3 召回率与速度的权衡

```
召回率-速度曲线 (典型HNSW, SIFT1M)

Recall@10
  100% │                                    ●(ef=512, 1000 QPS)
       │                              ●(ef=256, 2500 QPS)
   99% │                        ●(ef=128, 5000 QPS)
       │                  ●(ef=64, 8000 QPS)
   95% │            ●(ef=32, 12000 QPS)
       │      ●(ef=16, 16000 QPS)
   90% │●(ef=8, 20000 QPS)
       │
   80% │
       └──────────────────────────────────────────────► QPS
         2000    5000    10000    15000    20000

帕累托前沿: 速度每翻倍，召回率约下降 2-5 个百分点

企业选择建议:
- 搜索类应用 (语义搜索):     95-98% 召回率, ef=32-64
- 推荐类应用 (相关推荐):     99%+ 召回率, ef=128-256
- 消歧类应用 (精确匹配):     99.9%+ 召回率, ef=256-512
```

### 4.4 PQ 的率失真理论

```
乘积量化的率失真 (Rate-Distortion):

压缩率 R = m * log2(k) / d  bits per dimension
失真 D = E[||x - x'||^2]    均方量化误差

对于均匀分布的高维高斯数据:
  D ≈ σ^2 * (d/m) * k^{-2/(d/m)}

其中:
  σ^2 = 每维度的方差
  d/m = 子向量维度
  k   = 每子空间的聚类数

实际经验法则:
  - d/m = 8, k = 256 → 每个维度用 1 bit, 失真控制在 5-10%
  - d/m = 4, k = 256 → 每个维度用 2 bits, 失真控制在 1-3%
  - d/m = 2, k = 256 → 每个维度用 4 bits, 失真控制在 0.5-1.5%
```

## 6. 面试高频问题

### Q1: 向量数据库和传统数据库的核心区别是什么？

**答**: 三大核心区别：(1) **查询范式的不同** -- 传统数据库做精确匹配 (WHERE id=5)，向量数据库做模糊相似度搜索；(2) **索引结构的不同** -- 传统数据库使用 B-Tree/Hash 等精确索引，向量数据库使用 HNSW/IVF 等近似索引；(3) **评估标准的不同** -- 传统数据库追求精确结果和事务 ACID，向量数据库在召回率和响应延迟之间做权衡。

### Q2: HNSW 的三个核心参数 (M, efConstruction, efSearch) 如何影响性能？

**答**: M 影响图稠密度和内存占用，增大 M 提高召回率但增加内存和构建时间。efConstruction 影响索引构建质量和时间，增大后索引质量更高但构建更慢。efSearch 运行时参数，增大提高召回率但降低 QPS（线性关系：efSearch 翻倍，QPS 约减半，召回率提升 2-5 个百分点）。

### Q3: PQ 乘积量化的压缩原理是什么？召回率损失的主要原因？

**答**: PQ 将高维向量切分为 m 个低维子向量，对每个子空间独立进行 K-Means 聚类，用聚类中心 ID 替代原始子向量。压缩比可达 8x-32x。召回率损失的主要原因是量化误差 -- 原始向量与其 PQ 重建向量之间的差异，子空间独立导致忽略了跨子空间的相关性 (维度独立性假设)。

### Q4: 为什么 IVF 的 nprobe 增大会提高召回率但降低速度？

**答**: IVF 将搜索空间限制在 nprobe 个最近的聚类内。nprobe 增大意味着搜索更多的倒排列表，扫描更多的候选向量，精确最近邻落入搜索范围的概率增大（召回率提升），但距离计算量线性增长（速度下降）。典型权衡：nprobe 从 1 增到 16，召回率从 65% 提到 95%，但延迟从 0.5ms 增到 3ms。

### Q5: 向量数据库的 CAP 定理中属于哪一类？

**答**: 大多数向量数据库属于 CP 或可调节型。例如 Milvus 提供可调一致性 (Consistency Level)：STRONG (CP), BOUNDED (折中), EVENTUALLY (AP)。PGVector 依赖 PostgreSQL，默认 CP。Qdrant 使用 Raft，默认 CP。向量搜索通常偏向 CP (牺牲一些可用性换取搜索结果的一致性)。

### Q6: 1 亿条 768 维向量需要多少存储？

**答**: 原始 float32 向量: 1亿 * 768 * 4 字节 = 307GB。HNSW 索引附加: ~1.5x = 460GB (内存)。使用 PQ 压缩 (m=96): 1亿 * 96 字节 = 9.6GB + 码书约 0.3MB。使用标量量化 (SQ8): 1亿 * 768 字节 = 77GB。加上元数据、日志、副本后总存储通常为原始数据的 3-5 倍。

### Q7: 解释 DiskANN 如何利用 SSD 实现百亿级搜索？

**答**: DiskANN 基于 Vamana 图结构。核心技巧：(1) 图邻接表存储在 SSD 上，内存仅缓存热点节点；(2) 使用 PQ 编码在内存中进行快速近似距离计算，仅对候选向量进行 SSD 读取；(3) beam search 批量预取减少 I/O 次数；(4) 图构建时的 Robust Pruning 确保图直径在 O(logN) 量级，限制每次查询的磁盘 I/O 次数。单次查询通常只需 10-100 次 4KB 随机读。

### Q8: PGVector 适合什么样的场景？

**答**: 适合 (1) 已有 PostgreSQL 基础设施，不想引入新系统；(2) 向量数据需要与业务数据做复杂 JOIN；(3) 需要事务保证向量与元数据的一致性；(4) 向量规模在 5000 万以下；(5) 团队已经熟悉 PostgreSQL 运维。不适合 (1) 需要 GPU 加速；(2) 百亿级超大规模；(3) 极高 QPS (>10K) 的向量搜索。

### 7.2 性能优化清单

1. **选择合适索引**: HNSW 是通用最优选择，内存受限选 IVFPQ
2. **维度约减**: 使用 Embedding 模型的 Matryoshka 表示 (如 1536→256 维)
3. **量化加速**: 启用 SQ8 量化可将搜索速度提升 2x-4x，召回率损失 < 1%
4. **批量查询**: 将单条查询合并为批量 (batch_size=32/64)，利用 SIMD 加速
5. **连接池**: gRPC 连接复用，避免每次查询建立连接的开销
6. **读副本**: 查询密集型场景添加只读副本，读写分离
7. **索引预热**: 服务启动后预加载索引到内存，避免冷查询
8. **分片策略**: 按业务维度 (如 tenant_id) 分片，减少搜索范围

## 本章小结

向量数据库是 RAG 系统的基石。从 FAISS 的算法库到 Milvus 的分布式数据库，从 HNSW 的对数级搜索到 PQ 的极致压缩，向量数据库技术栈已经相当成熟。

**核心要点回顾**:

1. **向量数据库不可替代**: 传统数据库无法高效处理高维语义搜索，内存方案无法满足生产环境的持久化、并发、扩展需求。

2. **选型有据可依**: 小规模用 PGVector/Chroma，中规模用 Qdrant，大规模用 Milvus/Elasticsearch，超大规模需定制集群方案。

3. **索引算法是核心**: HNSW 是通用最优选择，IVFPQ 是内存受限下的最优方案，DiskANN 打开百亿级大门。

4. **参数调优需量化**: M=16-64, efConstruction=200-800, efSearch=32-512，需基于实际数据和业务目标做实验性调优。

5. **生产环境需完善**: 索引策略、监控告警、安全合规、灾备迁移 -- 任何一个环节的缺失都可能成为生产事故的起点。
